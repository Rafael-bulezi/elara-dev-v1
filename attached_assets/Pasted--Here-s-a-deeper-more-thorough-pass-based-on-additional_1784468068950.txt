### 1. Deeper Dive: Documented Ways Auth Breaks

I’ve expanded the list with more specific failure modes that appear repeatedly in real production reports.

#### A. Configuration & Environment Issues (Extremely Common)

| Issue | What Happens | Why | Sources |
|------|--------------|-----|---------|
| **Site URL still points to localhost** | OAuth / magic links / email confirmation redirect to localhost in production | Dashboard setting never updated after deploy | Afterbuild Labs, Supabase Redirect URLs docs, multiple rescue reports |
| **localhost vs 127.0.0.1 mismatch** | Session not detected after callback → redirect loop | Cookies are host-specific | Stack Overflow |
| **Cookie Domain / Subdomain problems** | After adding `www` or subdomain, logins break or rate-limit spikes | Old host-only cookies + new domain-scoped cookies conflict → dual refresh attempts | Medium post on subdomain cookie era conflicts |
| **Secure / SameSite flags wrong** | Cookies set but immediately dropped by browser | `Secure=true` on HTTP, or `SameSite=Strict` blocking OAuth cross-site return | Multiple production debugging guides |

#### B. Cookie & Session Handling Failures

| Issue | What Happens | Why | Sources |
|------|--------------|-----|---------|
| **Middleware does not return the response with updated cookies** | User appears logged out (especially after token refresh) | New tokens written to a response object that is later discarded on redirect | iamuvin.com, Supabase advanced guide |
| **httpOnly: true on Supabase cookies** | Client thinks user is logged out, server thinks logged in → loops | Browser client needs to read the cookies | DEV Community |
| **Large session cookies / chunking corruption** | Auth fails or headers exceed limits (Vercel/Cloudflare) | Big `user_metadata` forces multi-chunk cookies that can get corrupted | GitHub discussions + ssr issues |
| **Production vs Dev cookie behavior** | Works in `next dev`, breaks after `build` + `start` | Different cookie handling / middleware behavior | Multiple reports |

#### C. Token Refresh & Race Conditions (Very Painful in Production)

| Issue | What Happens | Why | Sources |
|------|--------------|-----|---------|
| **Single-use refresh tokens + middleware on every request** | Silent random logouts | Middleware burns the refresh token. If the browser doesn’t fully receive the new cookies (double-click, slow network, background tab), the old token is dead | [supabase/ssr #190](https://github.com/supabase/ssr/issues/190) |
| **Parallel requests during refresh** | Multiple components/middleware fire at once → only one refresh succeeds, others fail and force logout | Classic race condition with single-use tokens | Vercel Community, general SPA token refresh articles |
| **Stale refresh token sent from browser** | Server gets `invalid refresh token` | Client has not yet processed the new cookies | Supabase Advanced SSR guide |

#### D. getUser() / getSession() / getClaims() Confusion

| Issue | What Happens | Why | Sources |
|------|--------------|-----|---------|
| **Using getSession() for authorization** | Security risk (token can be spoofed) | Only reads local storage/cookies, does not verify with Auth server | Official docs + many warnings |
| **Calling getUser() too aggressively** | Rate limit errors or high latency | Every call hits the Auth server | Discord/AnswerOverflow reports, auth-js issues |
| **getClaims() vs getUser()** | Newer recommendation for performance | `getClaims()` does local JWT validation (faster), `getUser()` does full server check | Recent GitHub discussions + Medium explanations |

#### E. Other Notable Failures

- **Multi-session / local logout bugs** — logging out on one device can incorrectly kill sessions on others in some configurations ([supabase/auth #2036](https://github.com/supabase/auth/issues/2036)).
- **Missing `sub` claim** — happens when you accidentally pass an API key instead of a user JWT.
- **RLS + profile creation race** — signup succeeds but profile insert is blocked by RLS.
- **Clock skew** — short JWT expiry + device time drift causes immediate expiry (documented in Supabase sessions docs).
- **CDN caching Set-Cookie** — one user receives another user’s session (explicitly warned in Supabase advanced guide).

### 2. Deeper Dive: Scaling PostgreSQL to Millions of Users

Real-world patterns from OpenAI, Cloudflare, and others:

#### Proven Progression (Most Companies Follow This Order)

1. **Vertical scaling + good indexes** (first and biggest win)
2. **Connection pooling** (PgBouncer or Supabase Supavisor) — almost mandatory before high concurrency
3. **Read replicas** (offload SELECTs)
4. **Aggressive caching** (Redis / application-level)
5. **Partitioning** large tables
6. **Workload isolation** (separate clusters for different workloads)
7. **Sharding** (only when writes become the real bottleneck)

#### Key Real-World Examples

- **OpenAI (ChatGPT – 800M users)**: Single primary writer + ~50 read replicas globally. Heavy use of connection pooling, caching, and moving some write-heavy workloads elsewhere. They deliberately avoided early sharding on the main clusters.
- **Cloudflare**: Extremely high request volume with a relatively small number of Postgres clusters + strong connection pooling.
- **General consensus**: Most applications never need full sharding if they do connection pooling + read replicas + caching well.

#### Supabase-Specific Notes at Scale
- Use the **pooled connection string** (port 6543).
- Enable **Supavisor**.
- Watch RLS performance on very large tables (index the columns used in policies).
- Disk auto-scaling has limits and cooldowns.
- High RAM usage is common on small instances — optimize queries and connections first.

**Strong Scaling Sources**
- OpenAI Engineering: [Scaling PostgreSQL to power 800 million ChatGPT users](https://openai.com/index/scaling-postgresql/)
- Supabase Supavisor blog (1M connections)
- Various production guides on PgBouncer + read replicas
- Case studies showing the progression: single DB → pool → replicas → cache → (maybe) shard

**Database scaling** is much more predictable. You are very far from needing exotic solutions. Focus on indexes, connection pooling, and clean queries.
